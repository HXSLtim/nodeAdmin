# 文档维护机制

## 1. 维护原则

- 单一事实来源：同一主题仅保留一个主文档。
- 变更即更新：代码或方案变化必须同步文档。
- 可追溯：关键调整必须在 `decisionLog.md` 留痕。

## 2. 维护流程

1. 提出变更：描述背景、影响范围、目标状态。
2. 更新文档：修改对应主文档并补充关联引用。
3. 记录决策：新增或更新 `decisionLog.md` 条目。
4. 评审发布：由对应 owner 审核后标记生效。

## 3. 触发条件

- 新增服务或模块边界。
- 调整技术栈或中间件。
- 修改租户隔离、安全、合规策略。
- 变更路线图、里程碑或资源配置。

## 4. 评审清单

- 是否与 `platformSpec.md` 一致。
- 是否更新了引用关系（`docIndex.md`）。
- 是否更新了影响的计划文档（`roadmapPlan.md`）。
- 是否有可执行 owner 与生效日期。

## 5. 角色职责

- 架构负责人：架构类文档准确性。
- 交付负责人：计划与里程碑一致性。
- 项目负责人：治理规则与最终审批。

## 6. 更新频率

- 每周例行巡检一次。
- 里程碑开始前与结束后各一次。
